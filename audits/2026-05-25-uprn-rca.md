# OS Places UPRN regression — root cause analysis

**Date:** 2026-05-25 · **Author:** Phase 1 of UPRN-restore-and-observability prompt · **Scope:** diagnose only, no code changes

## TL;DR

**Root cause:** `lib/os-places.js :: lookupAddress` checks the circuit breaker *before* the cache. When the breaker is open, **the cache is never consulted** — even for addresses we've already resolved. The breaker trips on 3 consecutive 429s from OS Data Hub; the auto-reset after 10 min sends the very next probe live (bypassing 2,269 cached UPRNs); that probe also hits 429, re-trips, and the cycle becomes stable. **The system is stuck open and will never self-recover under load** until the cache-before-breaker order is corrected.

## Correction to the parent audit

Section 3.5 / 5.1 of `audit/2026-05-25-data-integrity-audit.md` stated `os_places_cache` had **0 rows**. That claim came from the Supabase `list_tables` MCP report, which returns approximate `pg_class.reltuples` stats that hadn't refreshed. A direct `COUNT(*)` shows **2,269 rows**, all positive hits, all written in a single 40-minute burst between 02:02 and 02:40 UTC on 2026-04-27 (a backfill run). The cache exists and is populated — it's just not being read. The parent-audit Section 5 finding will be amended after Phase 4.

## Findings

### 1. Cache writes ARE wired up

`lib/os-places.js` lines 213–227 (negative cache) and 250–266 (positive cache) both call `supabase.from('os_places_cache').upsert(...)`. There is no missing writer. The "no cache writes since 2026-04-27" symptom is a downstream effect of the breaker being permanently open — no API call ever completes, therefore no cache write ever fires.

### 2. The circuit breaker is in-memory only — and short-circuits the cache

```js
// lib/os-places.js line 129-131
if (osBreaker.isOpen()) {
  return { status: 'circuit_open' };       // <-- returned BEFORE the cache lookup
}

// ── Cache lookup ── (line 136)
if (supabase) { ... }
```

`isOpen()` (line 65) re-arms after `resetMs = 600000` (10 min) by zeroing the failure counter, which means the next call goes live. The state is module-singleton — never persisted, resets on process restart.

The defect: **cache lookups should not be gated on the breaker.** A cached UPRN is a static fact about a property; it doesn't depend on whether the upstream API is healthy. Returning `circuit_open` for an address we've already resolved discards work and amplifies the outage.

### 3. The 429 trigger

Of the lots processed in the last 60 days that DID escape the breaker briefly, the survival rate is overwhelmingly `api_error httpStatus=429` (rate-limit), not other failure modes (auth, timeout, network). The OS Data Hub Premium free tier (100k/mo) has both per-second AND aggregate rate limits; bursty enrichment (e.g. the 2026-04-04 spike of 3,137 new lots in one batch) likely tripped a per-minute or per-hour cap that the 100 ms `MIN_GAP_MS` doesn't defend against.

The combination is fatal:
1. Burst of new lots → cache miss (mostly new addresses)
2. Live API calls → 429s
3. 3 × 429 → breaker opens for 10 min
4. After 10 min, next call goes LIVE (cache check skipped) → another 429 → breaker re-opens
5. Steady-state: open forever

### 4. Daily distribution confirms the timeline

| Period | Pattern | What was happening |
|---|---|---|
| 2026-04-04 (3,137 new lots) | Mostly `no_stamp` + some `circuit_open` | Phase A code wasn't fully wired into all paths yet; the high-volume backfill was partially served |
| 2026-04-26 (Phase A landed) | `no_stamp` rate drops to near-zero | Every new lot now goes through OS Places |
| 2026-04-27 (cache backfill ran) | 2,269 rows written in 40 min | One-off populate — script ran outside the normal lookup path, so the breaker didn't matter |
| 2026-04-28 → 2026-05-02 | 14–26 `cache_hit`/day, plus mostly `circuit_open` | Cache lookups WORKING when the breaker happens to be momentarily closed; soon overwhelmed |
| 2026-05-03 → 2026-05-14 | 1–12 `cache_hit`/day, falling | Breaker now mostly stays open; only rare moments of cache lookup |
| 2026-05-15 → today | 0 `cache_hit`, 0 `ok`, all `circuit_open` (+ rare `api_error 429`) | Steady open state. **2,269 cached UPRNs are unreachable.** |

### 5. What ISN'T the cause

- **Not `OS_DATA_HUB_KEY` rotation.** The `api_error 429`s prove the key authenticates; rate-limit responses come back from a server that recognises the credential.
- **Not a missing writer.** Writers are wired correctly; they just don't fire because no API call completes.
- **Not a recent code regression.** No commit broke this — the cache-before-breaker defect was present from Phase A and only became visible once the live API started returning 429s reliably.
- **Not a cache-table wipe.** The cache has 2,269 rows; my Section 3 audit claim was wrong (see Correction above).
- **Not `os_places_cache` schema drift.** Schema matches the writer.

## What Phase 2 needs to do

1. **Fix the order in `lookupAddress`** — check the cache first, only fall through to the breaker check on cache miss. This is the structural fix and is the single most important change. (Note: pinned-contract rule — this is `lib/os-places.js`, not the `lib/types/lot.js` contract; safe to edit.)
2. **Reset the in-memory breaker** — happens automatically on process restart, but a one-shot `_resetCircuitForTest()` style hook lets us validate without a deploy.
3. **Single-lot smoke test** — pick one of the 20 most recent lots, call `lookupAddress(lot)` directly, confirm the response is either `cache_hit` (if address_key matches a cached row) or `ok` (if a live call succeeds). If neither, the API itself is broken and Phase 2 stops.
4. **Backfill 20 verification lots** — for each of the 20 most recently scraped lots, run `lookupAddress`, write the UPRN back to `lots`. Confirms forward-population. This bypasses the harness, runs sequentially with the 100ms gap, so won't itself trigger 429.
5. **Forward verification** — wait for the next scheduled scrape (or trigger one) and confirm new lots get `os_places.status = ok` or `cache_hit` in their manifest, and that `lots.uprn` is populated.

A secondary fix worth surfacing (but not making in Phase 2): the OS Data Hub rate-limit defence is too lax. `MIN_GAP_MS = 100` (= 600 RPM theoretical) leaves no headroom for the per-minute cap. A token bucket sized to the actual published limits would be more robust. **Surfacing as a recommendation, not making the change.**

## Verification predicate for Phase 4

After the fix:
- `enrichment_health` view should show: `status = ok|cache_hit` > 50% within 24 h of deploy; circuit_state transitions to `closed`.
- `lots` table: every new lot landed in the 24 h after deploy should have either a populated `uprn` (positive match), or `enrichment_manifest.os_places.status IN ('no_match','low_confidence','cache_hit_no_match')` (genuinely unmatchable address) — never `circuit_open`.

## Confidence

**High.** The defect is structural and visible in the code; the symptom distribution in the data exactly matches the predicted failure mode; the cache state confirms no writes have happened since the breaker latched. The only residual unknown is whether OS Data Hub will continue to 429 us under normal post-fix load — Phase 2 step 3 (single-lot smoke test) resolves that before any backfill.

# Plan 01-02: Stripe Hardening — Trial Abuse, Webhook Idempotency & Downgrade Logic

<objective>
Fix three Stripe-related bugs: (1) the `trial_used` flag is set but never checked, allowing trial re-use; (2) webhook events lack idempotency, causing duplicate processing on Stripe retries; (3) subscription cancellation triggers immediate downgrade instead of honouring `current_period_end`. These fixes prevent revenue leakage and ensure fair billing.
</objective>

<must_haves>
- A user whose email already has `trial_used: true` in the users table cannot obtain a second trial
- Duplicate webhook events (same `event.id`) are processed exactly once
- When a subscription is cancelled, the user retains premium access until `current_period_end`
- Payment failure (invoice.payment_failed) does not immediately downgrade — it sets a grace period or warning
- All changes follow existing patterns: Supabase queries, structured logging, try/catch error handling
</must_haves>

<task id="01-02-03" title="Add webhook event deduplication">
<action>
In `server.js` at ~line 1302, immediately after the Stripe signature verification succeeds and `event` is constructed, add idempotency check:

```javascript
// Idempotency: skip already-processed events
const { data: existing } = await supabase
  .from('processed_webhook_events')
  .select('event_id')
  .eq('event_id', event.id)
  .maybeSingle();

if (existing) {
  log(`Webhook event ${event.id} already processed, skipping`);
  return res.json({ received: true, duplicate: true });
}
```

Then at the END of the webhook handler (just before the final `res.json({ received: true })`), insert the event ID using upsert to handle race conditions where two retries arrive simultaneously:

```javascript
await supabase
  .from('processed_webhook_events')
  .upsert(
    { event_id: event.id, processed_at: new Date().toISOString() },
    { onConflict: 'event_id', ignoreDuplicates: true }
  );
```

<task id="01-02-04" title="Fix subscription.deleted to honour current_period_end">
<action>
In `server.js` at ~lines 1345-1362 (the `customer.subscription.deleted` handler):

**Before (current code):**
Sets `tier: 'free'`, `stripe_subscription_id: null`, `tier_expires_at: null` immediately.

**After:**
1. Extract `current_period_end` from the subscription object: `const periodEnd = new Date(sub.current_period_end * 1000).toISOString()`
2. If `current_period_end` is in the future, set `tier_expires_at: periodEnd` and keep `tier: 'premium'` — the existing expiry check in `validateUserFromReq` (~line 1625) will auto-downgrade when the date passes
3. If `current_period_end` is in the past or missing, downgrade immediately (keep current behavior)
4. Keep nulling `stripe_subscription_id` in both cases (subscription is deleted either way)
5. Log: `log('Subscription deleted for ' + email + ', premium until ' + periodEnd)`

```javascript
const periodEnd = sub.current_period_end
  ? new Date(sub.current_period_end * 1000)
  : null;

if (periodEnd && periodEnd > new Date()) {
  // Honour paid period
  await supabase.from('users').update({
    stripe_subscription_id: null,
    tier_expires_at: periodEnd.toISOString()
  }).eq('stripe_customer_id', sub.customer);
  log(`Subscription deleted, premium until ${periodEnd.toISOString()}`);
} else {
  // Period already ended, downgrade now
  await supabase.from('users').update({
    tier: 'free',
    stripe_subscription_id: null,
    tier_expires_at: null
  }).eq('stripe_customer_id', sub.customer);
  log('Subscription deleted, immediate downgrade');
}
```

<task id="01-02-05" title="Fix subscription.updated to not downgrade during active period">
<action>
In `server.js` at ~lines 1364-1378 (the `customer.subscription.updated` handler):

When status is `canceled` (user cancelled but period hasn't ended):
- Set `tier_expires_at` to `current_period_end` instead of immediate downgrade
- Keep `tier: 'premium'`

When status is `past_due`:
- Set a grace period (e.g., 3 days from now) as `tier_expires_at` instead of immediate downgrade
- Log a warning

When status is `unpaid`:
- Downgrade immediately (this means all retry attempts failed)

```javascript
case 'customer.subscription.updated': {
  const sub = event.data.object;
  if (sub.status === 'canceled') {
    const periodEnd = new Date(sub.current_period_end * 1000).toISOString();
    await supabase.from('users').update({
      tier_expires_at: periodEnd
    }).eq('stripe_customer_id', sub.customer);
    log(`Subscription canceled, premium until ${periodEnd}`);
  } else if (sub.status === 'past_due') {
    const grace = new Date(Date.now() + 3 * 24 * 60 * 60 * 1000).toISOString();
    await supabase.from('users').update({
      tier_expires_at: grace
    }).eq('stripe_customer_id', sub.customer);
    log(`Payment past_due, grace period until ${grace}`);
  } else if (sub.status === 'unpaid') {
    await supabase.from('users').update({
      tier: 'free',
      stripe_subscription_id: null,
      tier_expires_at: null
    }).eq('stripe_customer_id', sub.customer);
    log('Subscription unpaid, immediate downgrade');
  }
  break;
}
```
